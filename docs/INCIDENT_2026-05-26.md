# Incident 2026-05-26 — Disk-full cascade → Postgres crash → Redis Docker stuck

**Severity:** P1 — сервис фактически недоступен в течение нескольких часов.
**Detection:** видимо через несколько часов после disk-full начала ночи 25→26 мая. Замечено пользователем 2026-05-26 ~15:45 EEST.
**Root cause:** retention drain v3 (deployed 2026-05-25) не успевал догнать новые snapshots → диск ушёл в 0 байт за ночь → cascade crash Postgres → Redis (Docker) застрял в неконсистентном state с containerd.
**Outcome:** degraded recovery — Redis перезапущен без AOF/RDB (потеря in-memory state), workers рестартованы частично. Historical workers оставлены OFF до стабилизации.

## Root cause анализ

### Корневая причина: **disk-full cascade**

Архитектура **не может жить с 164 GB `api_payload_snapshot` на 436 GB диске без активного retention.** Каждый день:
- `api_payload_snapshot` растёт ~3.5 GB/day (new snapshots от live + historical workers)
- syslog растёт ~4 GB/day (Linux journald → syslog forward для 97 unit'ов)
- `pg_wal` колеблется 4-5 GB
- Buffer/cache used dynamically

**Свободного 5-10 GB запаса недостаточно** — любой пик (retention DELETE = WAL spike, BGSAVE Redis = double memory, system updates) выпихивает диск в 0%.

### Что усугубило

1. **Retention slow per tick** — 13 минут на полный tick из-за того что `live_snapshot_version_batch` и `live_state_history_batch` timeoutили (старый код без env-tunable). Я их **починил вчера**, но fix не успел дать effect до crash.
2. **`event_terminal_state` lock contention** — 597K rows locked. historical-hydrate workers тратили часы в TimeoutError loop. Каждый retry = WAL запись.
3. **Redis Docker не имел memory limit** (HostConfig.Memory=0) — RDB 11.91 GB загружался в peak ~24 GB, мог потенциально triggerить kernel OOM на host'е.
4. **Многоуровневая зависимость**: disk full → PG crash → workers crash → Redis OOM → workers in restart loop → load 63 → kernel scheduler stress → ещё больше delays.

### Почему Redis Docker застрял

containerd удерживал task `my-redis` в state "running" даже когда контейнер падал. dockerd пытался cleanup → "task must be stopped before deletion: running: failed precondition". В журнале docker.service эта ошибка повторяется каждые ~2 секунды. Восстановить можно ТОЛЬКО через `docker rm -f` или `systemctl restart docker`.

## Что было сделано

1. **+8 GB освобождено** через log cleanup (gzip syslog 4.3G→400M, journalctl vacuum 3.9G)
2. **+10 GB освобождено** через удаление orphan temp-*.rdb (failed BGSAVE attempts)
3. **+13 GB освобождено** через rename `appendonlydir` + `dump.rdb` (degraded recovery — потеря live state)
4. **Postgres** — сам recovered за 2 минуты после disk был >2G free
5. **Redis** — force kill + recreate с `--memory 16g --appendonly no --save ""`
6. **47 historical-* unit'ов оставлены OFF** для снижения нагрузки на event_terminal_state

## Уроки

### 🔴 Critical (срочно)

1. **Disk needs to be 1TB+** или отдельный volume под `/var/lib/postgresql`. Текущий 436G — недостаточно.
2. **Партиционировать `api_payload_snapshot`** по `fetched_at` (monthly). `DROP TABLE` старой партиции = мгновенно vs current slow DELETE-by-DELETE retention.
3. **Memory limit для Redis Docker container** — установлен `--memory 16g`. Если RDB вырастет — fail-fast вместо kernel OOM на host'е.
4. **Disk usage alert** — N1 monitoring должен alertить при <20G free. Сейчас alert приходит только когда уже катастрофа.

### 🟡 Major (на этой неделе)

5. **Переместить чужие контейнеры** (clawdbot, varmatch, yt_*) с этого сервера. /var/lib/docker = 65 GB.
6. **Journal compress** + `MaxFileSize=200M` в `/etc/systemd/journald.conf`. Syslog forward от 97 unit'ов = 4G/day, слишком много.
7. **Native Redis вместо Docker** или использовать `redis-server.service` (systemd) который никогда не упрётся в containerd consistency issue.
8. **AOF/RDB на проде**: рассмотреть `--appendonly no` или RDB только каждые 6h вместо continuous AOF — для рабочего dataset 12 GB AOF rewrite слишком тяжелый.

### 🟢 Minor

9. **Документировать procedure recovery** в OPERATIONS_RUNBOOK.md.
10. **Cascade restart script** должен сначала проверить диск >5G и Redis up.

## Файлы которые надо backup'нуть

Перед удалением (если решим выкинуть для освобождения 13G места):

- `/var/lib/docker/volumes/redis_data/_data/appendonlydir.bak/` — 7.9 GB старый AOF
- `/var/lib/docker/volumes/redis_data/_data/dump.rdb.bak` — 5.7 GB старый RDB

Эти файлы — снимок Redis state на момент crash. Содержат:
- All Redis streams (consumer groups, pending entries) — НО они уже бесполезны (workers заново создадут consumer groups через MKSTREAM)
- `zset:live:hot/warm/cold` — live state (восстановится через polling за 5-10 минут)
- `live:dispatch_metrics` — cumulative metrics (потеря не критична, перезапишется)
- `event_endpoint_negative_cache` (если был в Redis) — non-critical (recompute сам)
- Dedupe / lease / freshness keys — short TTL anyway, expire через минуты

**Рекомендация:** дать им полежать 24-48 часов на случай если что-то понадобится для forensics, потом удалить.
